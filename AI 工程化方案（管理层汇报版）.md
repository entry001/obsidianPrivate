# AI 工程化方案（管理层汇报版）

> 面向管理层的执行摘要版：用 AI 提升交付效率，但前提是把 AI 纳入工程化治理，而不是放任其“自由编码”。

---

## 一、这套方案要解决什么问题

当前 AI 已经可以显著提升编码速度，但在企业研发场景中，真正的瓶颈不在“写代码慢”，而在以下四点：

1. **需求到实现经常失真**：需求描述模糊，AI 和人都会各自理解，导致返工。
2. **代码快了，质量反而更难控**：AI 可在几分钟内产出大量代码，但缺陷、边界溢出、依赖失控会同步放大。
3. **跨系统协作风险升高**：主调度服务、前端、Electron、子系统共享数据库与协议，任何局部变更都可能引发系统级连锁问题。
4. **经验难沉淀**：如果每次都靠临时 Prompt 和个人经验驱动，团队无法形成稳定复用的工程资产。

因此，我们不是单纯“引入 AI 写代码”，而是建设一套 **AI 工程化操作系统**。

---

## 二、方案核心：两套引擎 + 一条流水线

### 1. OpenSpec：把“要做什么”定义清楚

OpenSpec 的目标，是把需求、接口、状态机、数据库、发布边界等内容，沉淀成机器可读、团队可审、跨角色共享的“真理源”。

它解决的是 **方向正确性问题**：
- 要改什么
- 不该改什么
- 会影响哪些仓库/接口/数据表
- 是否兼容旧版本
- 出问题如何回滚

### 2. Superpowers：让 AI 按工程纪律工作

Superpowers 的目标，是让 AI 在编码、测试、审计、发布时，不再“裸跑”，而是默认具备：
- 自动加载上下文
- 自动分析影响面
- 自动检查契约一致性
- 自动生成测试与验证
- 自动执行发布前门禁
- 自动把经验沉淀回规则与技能库

它解决的是 **执行稳定性问题**。

### 3. 一条端到端流水线

完整链路统一为：

**需求/缺陷 → Spec → 设计审计 → AI 实现 → PR → CI/CD → 发布/灰度 → 复盘沉淀**

这意味着 AI 不再是单点工具，而成为流水线中的“标准工位”。

---

## 三、为什么这套方案适合我们当前系统

我们的系统不是单仓库的小工具，而是由以下部分组成：
- 主调度服务
- 主调度 Web
- 看门狗 Web
- 看门狗 Electron
- 调度子系统
- PostgreSQL 共享数据层

这类系统的最大风险，不是单个功能点写不出来，而是：
- **跨仓库边界失控**
- **共享数据库被误写**
- **WebSocket / IPC / RPC 协议不一致**
- **热修复影响主链路**

所以，必须把“上下文、边界、契约、发布、回滚”一起纳入 AI 工程化治理。这正是本方案的价值所在。

---

## 四、组织层面的变化

这套方案不会减少管理，反而会让管理更清晰。

### 管理层关注的不是“谁写了多少代码”，而是：
- 是否把需求转成了可执行的 Spec
- 是否建立了自动化质量门禁
- 是否把事故和经验沉淀成规则资产
- 是否让跨仓库协作变得可控、可追踪、可回滚

### 角色分工将更聚焦：
- **架构/SE**：定义 OpenSpec、系统边界、协议与发布拓扑
- **平台/工具组**：维护 Superpowers、CI/CD、规则库与 Skill 库
- **开发**：把业务翻译成结构化场景，驱动 AI 生成并做首层审计
- **审计/值班**：负责发布门禁、运行验证、事故复盘与知识回流

换句话说：
**AI 负责加速产出，人类负责定义正确性、守住边界、经营资产。**

---

## 五、预期收益

### 1. 交付效率提升
- 需求转测试、测试转实现、接口转前后端联调都可以标准化加速
- 重复性代码与样板测试的生成效率显著提升

### 2. 质量更可控
- 通过 Lint / TypeCheck / Coverage / Security / Compatibility / Runtime Verify 等自动门禁，把问题挡在合入前
- 将“靠经验发现问题”转为“靠规则阻断问题”

### 3. 跨仓库协作更稳
- 所有变更先定义影响面，再编码
- 数据库、协议、事件、发布顺序都纳入统一审计

### 4. 团队资产可持续积累
- 规则、Skill、上下文、Runbook、复盘结论都沉淀为企业数字资产
- 新人和新项目可直接复用已有工程能力

## 六、建议管理层重点考核的 KPI

| 维度 | 建议指标 |
|------|----------|
| **效率** | 需求到 PR 周期、平均交付时长、测试生成时间 |
| **质量** | 缺陷逃逸率、回滚率、热修复占比、覆盖率达标率 |
| **稳定性** | 发布成功率、灰度失败率、线上事故恢复时长 |
| **资产化** | 新增 Skill 数量、规则复用率、复盘转规则闭环率 |
| **协作** | 跨仓库改动审计通过率、协议兼容性问题数量 |

---

## 七、实施建议：三步走

### 第一步：先立规矩
- 统一 Spec 模板
- 建立首批规则库与 Skill 库
- 配置最基本的 CI 门禁

### 第二步：选试点跑通
- 先挑 1 个核心后端模块、1 个前端页面、1 个数据库改动场景做试点
- 跑通“Spec → 审计 → 编码 → 测试 → 发布 → 复盘”闭环

### 第三步：全面复制
- 将试点经验升级为团队标准
- 全仓库推广统一的 Skill 与规则资产
- 建立月度复盘与规则迭代机制

---

## 八、管理层需要支持的三项决策

1. **确定 Spec 先行**：所有新需求与跨仓库改动必须先有结构化 Spec，再进入开发。
2. **授权平台化建设**：让工具/架构组有资源持续维护规则库、Skill 库、CI/CD 门禁与组件资产。
3. **建立复盘回流机制**：事故和高频问题必须回流成规则、Skill、测试或 Runbook，避免重复踩坑。

---

## 九、最终结论

这套方案的本质，不是“让 AI 写更多代码”，而是：

> **把 AI 纳入企业级研发流程，让需求定义、编码执行、质量审计、发布治理和知识沉淀形成闭环。**

如果说传统工程化解决的是“人如何稳定协作”，那么这套 AI 工程化方案解决的是：

> **人 + AI 如何在多仓库、多协议、多角色协作下，稳定、高速、可控地持续交付。**
