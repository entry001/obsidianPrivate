
AI 时代的 Spec 颗粒度有一个核心原则：**“声明业务意图与边界规则，隐藏具体技术实现细节。”** 它的颗粒度应当恰好卡在 **“能够唯一确定业务正确性”** 的层面上。

## 一、 Spec 的标准颗粒度：三层金字塔模型

在实际工程中，一个标准的需求 Spec 通常由以下三层颗粒度组成：

```
┌────────────────────────────────────────────────────────┐
│  1. 契约层 (接口与模型) -> 颗粒度：绝对精准 (100% 严密)   │
├────────────────────────────────────────────────────────┤
│  2. 行为层 (BDD 状态机) -> 颗粒度：业务闭环 (Given-When) │
├────────────────────────────────────────────────────────┤
│  3. 规则层 (上下文/禁忌) -> 颗粒度：全局/模块级约束      │
└────────────────────────────────────────────────────────┘
```

### 1. 契约层 (Contract/Schema)：绝对精准

- **内容：** 数据模型（Database Schema）和 API 接口定义（OpenAPI/YAML）。
    
- **颗粒度要求：** **100% 严密。** 字段名、数据类型（String, Int）、是否必填、枚举值（Enum）必须死死定死。
    
- **为什么：** 这是 AI 最不容易犯错的地方，只要 Schema 给得对，AI 生成的 DTO 和校验逻辑就是完美的。
    

### 2. 行为层 (Behavior/Flow)：业务闭环

- **内容：** 用 Gherkin 语法（Given-When-Then）描述的用户故事和核心业务流。
    
- **颗粒度要求：** **卡在“状态变更”和“副作用”上。** 不需要告诉 AI 怎么写 SQL，但必须告诉 AI 什么时候读、什么时候改、触发什么事件。
    

### 3. 规则层 (Rules/Constraints)：边界约束

- **内容：** 性能、安全、业务限制等硬性边界。
    
- **颗粒度要求：** **条件化、数字化。**（例如：“单次查询最大返回 100 条”、“重试次数限制为 3 次”）。
    

## 二、 怎么才能更加细致地描述需求的“业务流及边界”？

为了让 AI 拿到 Spec 后能 100% 顺从地编码，开发人员在编写 Spec 时，需要通过以下四个绝招来榨干描述的细致度：

### 1. 用“状态机思维”锁死业务流 (State Machine)

人类语言描述流程经常丢三落四，而大模型极度擅长理解**状态机**。描述复杂业务流时，直接给 AI 定义状态转移矩阵。

- **不要这样写：** “用户付款后变成已支付，如果退款就变成已退款。”
    
- **必须这样写：**
    
    Markdown
    
    ```
    【订单状态流转矩阵】
    - 当前状态 [PENDING_PAY] + 动作 [PAY_SUCCESS] -> 新状态 [PAID]
    - 当前状态 [PENDING_PAY] + 动作 [CANCEL] -> 新状态 [CANCELLED]
    - 当前状态 [PAID] + 动作 [REFUND_APPLY] -> 新状态 [REFUNDING]
    - 禁止越级流转：任何状态不可直接跳跃至 [FINISHED]，除非前置状态为 [SHIPPED]。
    ```
    

### 2. 穷举“正向、逆向、异常”三条路径 (The 3-Path Rule)

要想边界清晰，开发人员写 BDD 用例时，必须强迫自己和 AI 思考三个路径：

- **正向路径 (Happy Path)：** 最顺利的情况（输入正确，余额充足，扣单成功）。
    
- **逆向路径 (Unhappy Path)：** 业务允许的失败（余额不足，库存不够，优惠券过期）。
    
- **异常路径 (Edge/Exception Path)：** 技术和并发层面的边界（网络超时、高并发下的分布式锁竞争、恶意的非法参数）。
    

### 3. 针对边界值使用“等价类划分断言”

描述数量、金额、权限等边界时，不要给模糊的词汇（如“大额订单”），要给**绝对的断言数字**。

- **反面教材：** “如果是大额订单，需要额外审核。”
    
- **规范描述：**
    
    Markdown
    
    ```
    - Given: 订单金额 $X \in [0, 5000)$ -> Then: 自动放行，进入 PENDING_SHIP 状态。
    - Given: 订单金额 $X \in [5000, 50000]$ -> Then: 挂起订单，状态设为 PENDING_REVIEW，并触发审批工作流。
    - Given: 订单金额 $X > 50000$ 或 $X \le 0$ -> Then: 直接拦截，抛出 HTTP 400 (INVALID_AMOUNT)。
    ```
    

### 4. 显式声明“副作用”与“事务边界” (Side Effects)

AI 经常会漏掉看坑。你必须在 Spec 里明确圈出“不准漏掉的副作用”：

- **数据一致性：** “此接口涉及多表操作，必须在同一个数据库事务（Transaction）中运行。若扣减库存失败，必须整体回滚。”
    
- **外部通知：** “状态变为 PAID 后，**必须同步**发送一条 Kafka 消息到 `order-ev-topic`，且**必须异步**调用邮件通知服务。”
    

## 三、 实战：一个高质量、细颗粒度的 Spec 案例

以下是开发人员最终交付给 AI 工厂的标准化需求 Spec 示范（以“优惠券核销”为例）：

Markdown

```
# Spec: 优惠券核销业务边界规范

## 1. 契约定义 (Schema)
- API: POST /api/v1/coupons/redeem
- Request Body: { couponId: String(Required), orderId: String(Required) }
- Response: 200 { success: Boolean, discountAmount: Decimal } / 422 { code: String, message: String }

## 2. 核心业务流 (BDD Scenarios)

### 场景 1：正常核销 (Happy Path)
- Given: 用户拥有一张 ID 为 "CP-100" 的优惠券 (状态为 UNUSED，满 100 减 10)
- Given: 当前订单 "O-200" 的消费总额为 120 元
- When: 用户调用核销接口
- Then: 接口返回 200，`discountAmount` 动作为 10.00 元
- Then: 【副作用】数据库中 "CP-100" 的状态必须变更为 USED，且写入核销时间

### 场景 2：不可重入边界 (Edge Path - 并发防重)
- Given: 优惠券 "CP-100" 状态为 UNUSED
- When: 极短时间内（并发）收到两个完全相同的核销请求
- Then: 第一个请求成功处理
- Then: 第二个请求必须被分布式锁拦截，抛出 422 错误，错误码为 "COUPON_ALREADY_REDEEMED"
- Then: 确保同一张优惠券绝对不会被核销两次，数据库不允许出现负库存或双重扣减

## 3. 技术约束 (Constraints)
- Redis 锁：必须以 `lock:coupon:${couponId}` 作为 Key，超时时间设为 5 秒。
- 性能：核销操作整体耗时必须在 200ms 以内。
```

## 总结

在 AI 代码工厂里，**写 Spec 的过程，本质上就是开发人员在脑海中运行“高阶静态审计”的过程。** 你把这些边界、事务、并发锁用结构化的方式写出来（只需要花 10-15 分钟），Claude 就能拿着这份说明书，在几秒钟内为你交付一套毫无逻辑漏洞的高质量代码。
