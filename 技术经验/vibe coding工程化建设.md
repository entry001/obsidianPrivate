### 1. 建立极其严格的“自动化防御流水线”

既然代码是 AI 大量生成的，靠人工去一行行看是不现实的。必须用自动化工具来拦截垃圾代码：

- **超高覆盖率的单元测试：** 规定“Vibe 写出功能的同时，必须让 AI 把对应的 Unit Test 也写了”。
    
- **严苛的静态代码扫描：** 接入 SonarQube、Snyk 等工具，强制规范代码风格，自动拦截安全漏洞。
    
- **自动化 E2E（端到端）测试：** 确保 AI 改动 A 功能时，没有悄悄把 B 功能震碎。
    

### 2. 编写和维护“上下文白皮书” (Context Engine)

AI 编码的质量取决于它对你现有系统的了解程度。开发人员必须花大量时间去维护项目的**上下文（Context）**：

- 维护好项目的 `README.md`、`architecture.md` 以及专有的规则文件（如 `Cursor` 的 `.cursorrules`）。
    
- 明确定义技术栈、组件库使用规范、变量命名传统。让 AI 每次读取时，都能“入乡随俗”，而不是信马由缰地瞎写。
    

### 3. 实行“双人成行”或严格的“代码审计制”

Vibe Coding 容易让人产生“功能实现了就大功告成”的盲目乐观。

- **关键代码人工 Review：** 核心业务逻辑（如支付、核心数据修改）绝不允许 Vibe 完直接上线，必须由守门员进行人工 Code Review。
    
- **警惕幻觉引入：** 开发人员必须具备“不看懂代码绝不合入”的底线思维。
    

### 4. 转向“声明式”和“组合式”开发

开发人员的工作重点从“实现细节”变为“拼装积木”：

- 主导开发团队沉淀自己的“内部低代码/组件库”。
    
- 开发人员只需要告诉 AI：“请调用我们现有的 `AuthService` 和 `NotificationComponent` 来实现这个新页面。”而不是让 AI 每次都重新发明轮子。



## 一、 团队协作的核心：Git 工作流与“卡点”升级

在大团队中，必须剥夺开发人员直接修改主分支、甚至是普通开发分支代码的权力。所有人（包括 AI）必须在严格的 **分支保护策略** 下工作：

### 1. 团队标准的“无情流水线” (The Iron Pipeline)

开发人员通过 OMC 或 Cline 在本地生成了代码，但在他们提交 Pull Request (PR) 时，CI/CD 必须进行以下“非人化”的自动化卡点：

- **第一道关（代码静态检查）：** 无论 AI 写得多快，不符合 Prettier / ESLint / Checkstyle 规范，或者 SonarQube 扫描出圈复杂度超标、重复率超标，**一律秒拒**。
    
- **第二道关（自动化测试卡点）：** 运行 AI 编写的单测和 BDD 接口测试。如果测试通过率低于 100%，或者**代码覆盖率（Coverage）低于 85%**，PR 自动关闭。
    
- **第三道关（AI 审计合规性）：** 引入如 **CodeRabbit** 等企业级 AI Code Review 工具。它会读取全社统一的配置，自动检查：“此 PR 是否有未捕获的全局异常？是否私自引入了未登记的第三方依赖？”并生成审计报告。
    

### 2. 人类开发者的“黄金终审权”

在大团队里，开发人员的日常工作就是 **“看报告”**。

- 任何一个 PR 必须通过上面流水线的所有绿灯。
    
- 开发人员作为高级审计员，**只审核业务逻辑是否契合最初的设计、系统边界是否溢出**。通过后才允许合并（Merge）入研发主干。
    

## 二、 团队级的资产沉淀：统一“规范”与“上下文”

大团队共同协作，最怕 AI 各自发挥。必须把团队的架构和业务逻辑变成“企业数字资产”，让所有人的 AI 读同一本“圣旨”。

### 1. 建立共享的 `.omc/skills` 与企业级 Rules 库

大团队需要将所有规约代码化（Infrastructure as Rules），维护在一个中央公共仓库里：

- **公共规则仓（Monorepo Rules）：** 统一编写和更新团队的 `.cursorrules`、`.clinerules` 以及 OMC 的 `skills`。
    
- **分发机制：** 团队成员本地的项目通过 Git Submodule（子模块）或每次 `npm run dev` 时自动从中央仓拉取最新的 AI 规约。确保整个团队 50 号人使用的 Claude 拥有完全一致的编码偏好、技术栈选择和安全禁忌。
    

### 2. 沉淀企业级“黑盒组件库”（Domain LEGOs）

如果把业务细节全部暴露给 Claude，它必失控。

- 资深开发和架构师的核心 KPI，是去封装、沉淀内部的“乐高积木块”（如：统一的分页插件、标准的对账单中间件、CrownBio 专用的文件解析器）。
    
- **通知 AI 禁用底层代码：** 在共享规则里写明：_“团队已封装 `StandardHttp`，禁止直接使用 `axios` 或 `fetch`。”_ 这样大团队的代码风格就能保持高度一致。
    

## 三、 团队人员的角色重新调配 (Org Chart)

在全员 Vibe Coding 的大团队中，不能再按照传统的“前端组”、“后端组”来划分，而应该按 **流水线的上下游** 来重组：

```
                    ┌──────────────────────────┐
                    │    业务设计组 (人类)      │ ──> 纯写 BDD (Given-When-Then), 设计业务流
                    └──────────────────────────┘
                                 │
                                 ▼
┌────────────────────────────────────────────────────────────────────────┐
│                        AI 自动化编码工厂 (OMC / Claude)                  │
└────────────────────────────────────────────────────────────────────────┘
                                 │
                                 ▼
                    ┌──────────────────────────┐
                    │    质量审计组 (人类)      │ ──> 看 CI 测试报告、看 AI 审计、把关系统边界
                    └──────────────────────────┘
                                 ▲
                                 │ 支撑
                    ┌──────────────────────────┐
                    │    工具架构组 (人类)      │ ──> 维护底层组件库、更新全局 .clinerules / skills
                    └──────────────────────────┘
```

1. **业务设计组 (Business Designers)：**
    
    - 由原中高级开发转型。他们 100% 的精力都在跟产品经理对齐业务，用最严密的 Markdown / BDD 语法把复杂的业务状态机设计出来。
        
2. **工具与架构组 (Platform/Infra Engineers)：**
    
    - 团队里技术最强的一小撮人。他们**不碰具体业务**，专职维护流水线（CI/CD）、打磨企业专属规则库（Skills）、封装高内聚的基础组件。
        
3. **质量与审计组 (Auditors / Quality Guards)：**
    
    - 负责守门。盯着代码覆盖率报告、线上 OpenTelemetry 链路追踪和日志。只要有故障，负责通过链路图定位到问题分支，打回给对应的开发让 AI 重新 Vibe 修复。
        

## 四、 大团队落地的“三步走”策略

让大团队一夜之间全面转型是不切实际的，会引发巨大的组织混乱。建议采取**渐进式落地**：

- **第 1 周：规范先行（定义圣旨）**
    
    架构师牵头，先写出一版最严厉的全局规则文件（`.clinerules` / `.cursorrules`），并在 CI 里配好最基础的代码格式检查（Lint）和覆盖率卡点。
    
- **第 2-3 周：试点突击（打样示范）**
    
    选 1-2 个**边界清晰的新功能模块**或者一个 3-5 人的敏捷小组，作为试点。让他们全面采用 OMC 的 `Team` 或 `Autopilot` 模式，按照“人写 BDD $\rightarrow$ AI TDD 闭环 $\rightarrow$ 人做审计”的流程跑通一个小版本。
    
- **第 4 周起：全面复制（标准外推）**
    
    把试点的成功经验（包括踩坑记录、更完善的 Prompt 模板）作为案例，在全团队推广，并将原团队拆分为“设计、审计、架构”三种角色。
