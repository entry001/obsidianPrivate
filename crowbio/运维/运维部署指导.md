## 3. 整体部署架构

### 3.1 架构说明

- 对外统一入口通过 **CloudFront** 暴露。
- CloudFront 回源到 **ALB**。
- ALB 将请求转发给 **业务后端**。
- 业务后端通过内网服务发现或内网地址调用 **AI 中台**。
- 业务后端同时依赖 **PostgreSQL、Redis/Valkey、S3、OnlyOffice**。
- AI 中台依赖 **Azure OpenAI**，并可按需接入 **Bedrock**。

## 4. 依赖结构

### 4.1 分层依赖

#### 1）接入层

- **CloudFront**：对外统一域名入口、HTTPS、缓存加速
- **ALB**：将流量转发给业务服务

#### 2）应用层

- **业务后端 `reportgen-backend`**
  - 对外提供业务 API
  - 对接 SSO
  - 调用 AI 中台聊天接口
  - 调用 OnlyOffice 文档服务
  - 访问 PostgreSQL、Redis、S3
- **AI 中台 `aiagent`**
  - 提供 `/api/v1/chat/sse`
  - 通过 `BACKEND_BASE_URL` 反向调用业务后端工具接口
  - 对接 Azure OpenAI
  - 接入 Bedrock 短期记忆
- **OnlyOffice**
  - 通过统一域名下的 `/onlyoffice/` 路径提供服务
  - 与业务后端回调地址互通

#### 3）数据与能力层

- **PostgreSQL / RDS**：业务主数据
- **Redis / Valkey**：缓存、会话或加速场景
- **S3**：静态资源、报告文件、附件存储
- **Azure OpenAI**：LLM 能力来源
- **Bedrock**：记忆或扩展能力

## 5. 部署方式

### 5.1 总体部署顺序

建议按以下顺序交付：

1. 准备 AWS 账号、网络、域名、证书
2. 创建或确认 S3、RDS、Redis/Valkey 等基础资源
3. 创建 ECR 仓库并推送镜像
4. 创建/更新 ECS Task Definition
5. 发布 `reportgen-backend`
6. 发布 `aiagent`
7. 发布 `onlyoffice-documentserver`
8. 配置 CloudFront 与 ALB 转发
9. 执行联调与验收

### 5.2 AWS 登录

```bash
aws sso login --profile cb-ps-ai-it-rw-838261167460
```

### 5.3 业务后端部署

业务后端是基于 **Spring Boot 3 + Maven** 的多模块服务。

#### 构建方式

```bash
mvn clean package -DskipTests
```

默认产物：

```text
crownBio-admin/target/crownBio-admin.jar
```

#### 本地/服务器启动示例

```bash
java -jar crownBio-admin/target/crownBio-admin.jar --server.port=8080 --spring.profiles.active=prod
```

#### Docker 镜像构建建议

```bash
docker build -f Dockerfile.prod --build-arg JAR_FILE=crownBio-admin/target/crownBio-admin.jar -t crownbio-backend:prod .
```

#### 推送到 ECR 的典型流程

```bash
aws ecr get-login-password --region ap-east-1 --profile cb-ps-ai-it-rw-838261167460 | docker login --username AWS --password-stdin 838261167460.dkr.ecr.ap-east-1.amazonaws.com

docker tag crownbio-backend:prod 838261167460.dkr.ecr.ap-east-1.amazonaws.com/crowbio/server:<tag>
docker push 838261167460.dkr.ecr.ap-east-1.amazonaws.com/crowbio/server:<tag>
```

### 5.4 AI 中台部署

AI 中台基于 **FastAPI + Strands SDK**。

#### ECS 生产部署方式

可使用项目内现有脚本或标准 Docker 流程：

```bash
docker build -t crowbio/aiagent:<tag> .
docker tag crowbio/aiagent:<tag> 838261167460.dkr.ecr.ap-east-1.amazonaws.com/crowbio/aiagent:<tag>
docker push 838261167460.dkr.ecr.ap-east-1.amazonaws.com/crowbio/aiagent:<tag>
```

### 5.5 OnlyOffice 部署

OnlyOffice 当前以独立容器方式部署在 ECS/Fargate：

- 容器端口：`80`
- 通过统一域名下 `/onlyoffice/` 路径暴露
- 需要确保与业务后端回调地址可互通

如使用自定义镜像版本，流程与其他 ECS 容器一致：

1. 构建或准备镜像
2. 推送到 ECR
3. 更新 Task Definition
4. 触发 ECS 滚动发布

### 5.6 ECS 服务更新

#### 常用查询命令

```bash
aws ecs list-tasks --cluster vibrant-zebra-f7l8x2 --region ap-east-1
aws ecs describe-services --cluster vibrant-zebra-f7l8x2 --services <service-name> --region ap-east-1
```

#### 强制重新部署

```bash
aws ecs update-service --cluster vibrant-zebra-f7l8x2 --service <service-name> --force-new-deployment --region ap-east-1
```

#### 启用并进入容器执行命令

```bash
aws ecs update-service --cluster vibrant-zebra-f7l8x2 --service <service-name> --enable-execute-command --region ap-east-1
aws ecs execute-command --cluster vibrant-zebra-f7l8x2 --task <task-id> --container <container-name> --interactive --command "/bin/bash"
```

### 5.7 测试/跳板机场景部署

除 ECS 之外，目录中的文档还提供了基于 **SSH/SCP 上传包到 EC2** 的部署方式，适用于测试或临时环境。

#### 后端

- 本地打包 Jar
- 上传到远端服务器
- 执行远程更新脚本

#### AI 中台

- 本地打包 tar.gz
- 通过 `scp` 上传到目标主机
- 解压后执行 `start.sh`/`stop.sh`

该方式适合快速联调，但正式交付建议以 **ECS + ECR + Task Definition** 为主。

## 6. 运维注意事项

1. **敏感信息不要继续保存在文档明文中**，建议统一迁移到 Secrets Manager / SSM。
2. 业务后端当前存在 `FILE_STORAGE_TYPE=local` 与本地目录 `/data/uploadPath` 配置；若 ECS Task 重建后仍需保留文件，建议评估改为 **S3 或 EFS** 持久化方案。
3. 若使用多区域（如 `ap-east-1` 与 `us-east-1`），应明确区分：
   - 数据库实例
   - Redis/Valkey 实例
   - S3 Bucket
   - 前端静态资源桶
4. CloudFront、ALB、OnlyOffice 回调路径应在上线前做一次端到端联调，避免跨域、回源、路径重写问题。
5. ECS 服务间调用建议固定使用内网服务名或服务发现名称，避免手工维护 IP。

## 7. 交付结论

当前项目已具备完整的交付骨架：

- **接入层**：CloudFront + ALB
- **应用层**：业务后端 + AI 中台 + OnlyOffice
- **数据层**：PostgreSQL + Redis/Valkey + S3
- **能力层**：Azure OpenAI + Bedrock

推荐正式交付方式为：

1. **镜像统一推送 ECR**
2. **ECS/Fargate 统一托管服务**
3. **CloudFront + ALB 对外提供访问**


以上方式最符合当前目录中已有的资源结构与运维脚本设计。
